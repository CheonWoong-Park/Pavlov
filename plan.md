# 연구계획서 (최종)

## 디퓨전–자기회귀 하이브리드 디컴파일: 구조 골격 복원에 대한 생성 패러다임 통제 비교

---

## 1. 개요

디컴파일을 구조 골격 복원과 토큰 충전의 두 단계로 분해하고, 골격 복원 단계를 디퓨전 언어모델과 자기회귀(AR) 언어모델로 각각 학습하여 비교한다. 두 모델은 동일한 백본(Qwen2.5-Coder-7B)에서 파생된 동일 크기의 쌍이므로, 성능 차이는 생성 패러다임의 차이로 귀속된다. 목표 산출물은 워크숍/단편 논문 1편이며, 전체 실험은 RTX 4090 24GB 단일 데스크탑에서 수행한다. 일정은 8주.

---

## 2. 배경과 문제

디컴파일은 컴파일된 바이너리를 사람이 읽을 수 있는 소스코드로 복원하는 작업으로, 악성코드 분석·레거시 유지보수·취약점 연구의 기반 기술이다. Ghidra 등 전통 도구의 의사코드 출력은 타입과 식별자가 손실되어 가독성이 낮고, 이를 LLM으로 개선하는 연구(LLM4Decompile, SK²Decompile, ReF Decompile 등)가 활발하다. 그러나 기존 LLM 디컴파일러는 전부 토큰을 좌에서 우로 생성하는 자기회귀 방식이다.

코드 복원에는 전역 제약이 많다. 함수 후반부의 변수 사용 방식이 전반부의 타입 선언을 결정하고, 중괄호 짝과 분기 구조는 함수 전체에 걸쳐 일관되어야 한다. 단방향 생성은 뒤를 보기 전에 앞을 확정해야 하므로 이런 제약에 원리적으로 불리하다. 반면 디퓨전 언어모델은 시퀀스 전체를 양방향으로 보며 여러 스텝에 걸쳐 정제하는 방식으로, 전역 구조를 잡는 데 유리할 수 있으나 세부 토큰 품질과 추론 비용에서는 자기회귀에 밀리는 경우가 많다.

디퓨전 언어모델을 디컴파일에 적용한 연구는 현재 공백 상태다. 본 연구는 이 공백을 "디퓨전이 잘할 만한 하위 과제(구조 복원)만 골라 맡기고, 그 효과를 통제된 조건에서 측정한다"는 방식으로 메운다.

---

## 3. 연구 질문과 가설

**RQ1.** 동일 백본·동일 크기·동일 학습 예산 조건에서, 디퓨전 골격 복원기는 자기회귀 골격 복원기 대비 골격 품질(구문 유효성, 구조 일치)과 최종 디컴파일 품질(재실행성, 가독성)에서 어떤 차이를 보이는가?

**RQ2.** 2단계 분해(골격 복원 → 토큰 충전) 자체가 단일 단계 직접 디컴파일 대비 이득이 있는가?

**RQ3.** 디퓨전과 자기회귀의 격차는 함수 길이, 중첩 깊이, 컴파일러 최적화 수준에 따라 어떻게 달라지는가?

**가설.** 디퓨전의 양방향 문맥과 반복 정제는 전역 제약이 빡빡한 조건 — 긴 함수, 깊은 중첩, 고최적화(O2/O3) 바이너리 — 에서 구조 유효율의 우위로 나타나고, 이것이 최종 재실행성으로 전이된다. 짧고 단순한 함수에서는 격차가 작거나 없을 것이다.

가설이 기각되더라도 단계별 지표 덕분에 어느 단계에서 왜 밀렸는지 진단할 수 있으므로, 결과 방향과 무관하게 "디퓨전 LLM이 디컴파일에서 언제·왜 다르게 동작하는가"에 대한 첫 통제 분석으로 성립한다.

## 4. 관련 연구와 본 연구의 위치

| 분야 | 대표 연구 | 본 연구와의 관계 |
|---|---|---|
| 자기회귀 LLM 디컴파일 | LLM4Decompile, ReF Decompile, Idioms | 베이스라인 패러다임. 평가 프로토콜(재실행성)과 공개 보고치 차용 |
| 2단계 분해 디컴파일 | SK²Decompile (골격→피부) | 분해 구조 차용. 골격 단계의 패러다임 교체가 본 연구의 차별점 |
| 코드용 디퓨전 LLM | DiffuCoder, Dream-Coder, Stable-DiffCoder, TreeDiff | 디퓨전 팔의 베이스 모델 및 학습 레시피 차용. 디컴파일 적용은 본 연구가 처음 |
| 데이터·벤치마크 | Decompile-Bench (NeurIPS 2025 D&B) | 데이터 원천. 신규 수집 불필요 (5.4절) |

## 5. 방법

### 5.1 전체 파이프라인

```
decompile-bench 공개 바이너리(디버그 정보 포함)
        → [Ghidra headless (analyzeHeadless)] → 의사코드 (함수 단위)
        → [1단계: 골격 복원기(학습 대상)] → 익명화 구조 골격
        → [2단계: 토큰 충전기(동결)] → 최종 소스코드
```

주의: 데이터셋의 `asm` 필드는 어셈블리이며 본 연구의 입력이 아니다. 본 연구의 입력은 Ghidra 의사코드이므로, 데이터셋과 함께 공개된 바이너리에 Ghidra headless를 직접 실행하여 함수 단위 의사코드를 추출한다. 추출된 의사코드는 데이터셋의 함수명(demangled name) 기준으로 정답 소스와 매칭한다.

### 5.2 골격 표현

tree-sitter(tree-sitter-c)로 소스를 파싱하여 식별자를 `VAR_n`/`FUNC_n`으로, 리터럴을 타입별 placeholder(`INT_LIT`, `STR_LIT` 등)로 치환한 텍스트를 골격으로 정의한다. 제어흐름, 중첩, 타입 골조, 연산자 구조는 보존된다. CFG/AST를 직접 생성 대상으로 삼지 않는 이유: 그래프 생성은 별도 연구 영역이며, 텍스트 골격은 기존 masked diffusion 학습 레시피를 그대로 적용할 수 있고 SK²Decompile의 중간 표현과도 비교 가능하다. AST/CFG는 생성 표현이 아니라 평가 지표로 사용한다.

### 5.3 모델과 학습

| 항목 | 디퓨전 팔 (D) | 자기회귀 팔 (A) |
|---|---|---|
| 베이스 | DiffuCoder-7B | Qwen2.5-Coder-7B |
| 손실 | masked diffusion loss (공개 레시피) | 표준 cross-entropy |
| 공통 설정 | bf16 베이스 + 16-bit LoRA, r=32, alpha=64, attention+MLP 전층, gradient checkpointing, AdamW, seq ≤ 4096, micro-batch 1–2 + gradient accumulation | 동일 |
| 시드 | 2개 | 2개 |

DiffuCoder-7B는 Qwen2.5-Coder-7B를 디퓨전으로 적응시킨 모델이므로 두 팔은 백본·크기·계열이 동일하다. 학습 데이터·에폭·어댑터 예산도 동일하게 맞춘다. 따라서 결과 차이의 귀속 대상은 생성 패러다임 하나로 좁혀진다.

2단계 충전기는 Qwen2.5-Coder-7B-Instruct(bf16, 동결)를 사용하며, 골격 보존 제약을 명시한 프롬프트로 식별자·리터럴만 채우게 한다. 학습 변수를 1단계에만 격리하기 위해 충전기는 학습하지 않는다.

예상 메모리: 학습 17–19GB, 추론 14–16GB (24GB 내 안착). 학습 시간: 팔당 시드당 약 1–1.5일, 총 4런 4–6일.

### 5.4 데이터

**출처 (전부 공개, 신규 수집 없음).**

| 자원 | 위치 | 내용 |
|---|---|---|
| 학습 원천 | Hugging Face `LLM4Binary/decompile-bench` | 허가형 라이선스 GitHub 프로젝트에서 컴파일한 바이너리-소스 함수 쌍 약 200만 개. 레코드 필드: 함수명(demangled)·소스코드·어셈블리·파일 경로. 약 8.46GB. 의사코드 추출용 바이너리(디버그 정보 포함) 별도 공개 |
| 평가 원천 | Hugging Face `LLM4Binary/decompile-eval` (Decompile-Bench-Eval) | (1) HumanEval·MBPP에서 제작한 바이너리 — 테스트 케이스 보유, (2) 데이터 누출 방지를 위해 2025년 이후 공개된 GitHub 저장소 컴파일분 |
| 스크립트 | GitHub `albertan017/LLM4Decompile` | 평가 하니스, Ghidra 관련 스크립트, 기존 모델 보고치 |

**언어 필터: C 함수만 사용.** decompile-bench는 C와 C++을 모두 포함하지만, 가독성 지표 R2I가 C 전용이고 익명화 파서도 단일 언어(tree-sitter-c)로 통일하는 것이 워크숍 범위에 적절하므로 C로 한정한다. 이 제한은 논문 한계 절에 명시한다.

**구축.**
- 학습: 100,000쌍. 바이너리 → Ghidra 의사코드 추출 → 함수명 매칭 → 소스 익명화(5.2)로 (의사코드, 골격) 쌍 생성. 최적화 레벨 O0–O3 균형, 시퀀스 합 4096 토큰 이하. Ghidra 추출·매칭 수율 손실을 감안해 원천에서 1.5–2배 여유 추출.
- 평가: 600 함수, decompile-eval에서 층화 추출(최적화 레벨 × 함수 길이 × 중첩 깊이, 256토큰 경계 양쪽 포함). 재실행성은 테스트 케이스가 있는 HumanEval·MBPP 분할에서, 누출 안전 분석(골격 지표·가독성·컴파일 성공률)은 GitHub2025 분할에서 측정하고 분할별로 분리 보고한다.

## 6. 실험 설계

### 6.1 비교 조건

| 조건 | 1단계 | 2단계 | 답하는 질문 |
|---|---|---|---|
| B0 | 없음 (Ghidra 원본) | 없음 | 하한 기준 |
| B1 | 없음 | Instruct가 의사코드→소스 직접 생성 | 분해 자체의 이득 (RQ2) |
| A | AR 골격 복원기 (×2 시드) | 동결 충전기 | RQ1 비교군 |
| D | 디퓨전 골격 복원기 (×2 시드) | 동결 충전기 | RQ1 실험군 |
| B2 (선택) | DiffusionGemma-26B-A4B 4-bit 제로샷 | 동결 충전기 | 부록 참고점. 일정 여유 시에만 |

### 6.2 평가 지표

**골격 단독 (1단계 품질)**
- 파싱 성공률: tree-sitter 파싱 통과 비율
- AST 편집 거리: 정답 골격 대비 트리 편집 거리
- 제어흐름 골조 일치율: 분기·루프 구조의 그래프 수준 일치

**End-to-end (최종 품질)**
- Re-executability: GCC 재컴파일 후 원본 테스트 통과율 (HumanEval·MBPP 분할)
- R2I: 디컴파일 전용 상대 가독성 지표, C 전용 (BLEU 류 텍스트 유사도는 디컴파일 평가에 부적절하므로 배제)
- 골격 위반율: 충전기 출력이 입력 골격의 구조를 벗어난 비율 (분해 파이프라인의 건전성 점검)

**비용**
- 학습 GPU 시간, 추론 지연(디퓨전 denoising 스텝 수 명시)

**측정 타당성 점검 (sanity check).** B0·B1의 측정치를 Decompile-Bench 논문 공개 보고치와 대조하여 하니스의 정상 동작을 검증한다. 기준점 예: 전통 도구(IDA)의 R2I는 약 40 수준, LLM 기반 디컴파일러는 약 60–70 수준, LLM4Decompile-DCBench-6.7b의 재실행성은 약 39% 수준으로 보고되어 있다. 우리 B0·B1 측정치가 이 자릿수에서 크게 벗어나면 하니스를 점검 후 본실험에 진입한다.

### 6.3 통계 처리

- A vs D 차이에 paired bootstrap 유의성 검정 (함수 단위 쌍대 비교)
- 시드 간 분산을 모든 표에 병기

## 7. 분석 계획

1. **분해 분석 (RQ3, 핵심 그림).** A vs D 격차를 함수 길이, 중첩 깊이, 최적화 레벨별로 분해. 가설대로면 전역 제약이 강해질수록 격차가 벌어지는 단조 패턴이 나타나야 한다.
2. **단계 귀속.** 골격 단독 지표와 end-to-end 지표를 교차하여, 최종 품질 차이가 1단계에서 발생했는지 2단계 충전에서 희석/증폭됐는지 분리.
3. **정성 사례.** denoising 중간 스냅숏 3건을 추출하여 반복 정제가 실제로 어떤 구조 오류(괄호 불일치, 중첩 꼬임 등)를 고치는지 제시.
4. **실패 분석.** D가 A에 밀리는 구간이 있다면 그 구간의 공통 특성(예: 짧은 함수, 특정 제어 패턴)을 기술.

## 9. 리스크와 대응

| 리스크 | 가능성 | 대응 |
|---|---|---|
| Ghidra 추출·매칭 수율 미달 | 중 | 1.5–2배 여유 추출, 매칭 기준 단계적 완화, 수율을 G0 게이트로 조기 확인 |
| 컴파일·테스트 하니스 공수 초과 | 중 | 2–3주차 최우선 배치. albertan017/LLM4Decompile 공개 평가 스크립트 최대 재사용 |
| 디퓨전 팔의 골격 품질 미달 (G3) | 중 | 골격 어휘 축소, denoising 스텝 수 상향. 그래도 미달 시 "디퓨전이 이 과제에서 실패하는 지점" 자체를 분석 결과로 전환 |
| 충전기의 골격 위반 | 중 | 프롬프트 제약 강화 + 골격 위반율 상시 감시, 위반 사례 별도 집계 후 한계 절 보고 |
| LoRA가 디퓨전 적응에 불충분 | 저–중 | 양 팔 동일 어댑터 예산이므로 비교 공정성 유지. "동일 어댑터 예산 하 비교"로 명시 |
| 디퓨전 추론 비용 과다 | 저 | 골격이 짧아 denoising 토큰 수 자체가 적음. 스텝 수–품질 곡선 보고로 투명화 |
